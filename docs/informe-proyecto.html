1. Resumen Ejecutivo

Se construyó un sitio web completo para el Azul Horizonte Boutique Hotel (Playa Champerico, Retalhuleu) que integra un agente de IA conversacional (Claude) con un sistema automatizado de reservaciones orquestado por Zapier y n8n.

El proyecto cubre tres dimensiones principales:

  1. Sitio web público — páginas estáticas de presentación del hotel, habitaciones, servicios y preguntas frecuentes.
  2. Agente de IA — chat widget embebido que responde preguntas con contexto del hotel (RAG) y captura reservaciones en tiempo real verificando disponibilidad en Google Calendar.
  3. Automatización post-reserva — flujos en Zapier y n8n que envían correos de confirmación, recordatorios 24 h y 2 h antes del check-in, gestionan cancelaciones/modificaciones y generan reportes mensuales al administrador.

3.3 Agente de IA (Claude)

El agente corre en el servidor. Cada mensaje del usuario pasa por dos pasos antes de llegar a Claude:

  1. RAG (Retrieval-Augmented Generation): rag.ts busca por keywords en los archivos .md del hotel (habitaciones, políticas, servicios, FAQ) y adjunta los fragmentos relevantes al system prompt.
  2. Llamada a Claude API: chatFn.ts envía el historial completo de la conversación más el contexto RAG. Claude puede responder en texto o invocar el tool show_booking_form.

El agente tiene definido un único tool:

{
  name: "show_booking_form",
  description: "Muestra el formulario de reservación al cliente cuando está listo para reservar",
  input_schema: { type: "object", properties: {}, required: [] }
}

Cuando Claude llama este tool, el frontend detecta la tool_use en la respuesta y renderiza el componente <BookingForm /> dentro del panel del chat.

3.4 Módulo de Disponibilidad

google-calendar.ts se comunica con la Google Calendar API usando una Service Account (credenciales en variable de entorno). La función isRoomAvailable(roomType, checkIn, checkOut) lista los eventos en el calendario único "hotel" cuyo título contenga el código de la suite pedida y verifica que no exista traslape de fechas.

3.5 Procesamiento de Reservaciones (Backend)

bookFn.ts (server function) ejecuta el siguiente flujo:

  1. Recibe el payload del formulario: nombre, email, teléfono, check-in, check-out, tipo de habitación, huéspedes.
  2. Genera un bookingId (UUID).
  3. Llama isRoomAvailable() → Google Calendar API.
  4. Si disponible: llama sendToZapier() con todos los datos + UUID. Retorna { status: "confirmed", bookingId }.
  5. Si no disponible: retorna { status: "unavailable" } sin llamar Zapier.

5. Configuración de Zapier

Trigger: Webhooks by Zapier → Catch Hook. El backend llama este webhook únicamente cuando la disponibilidad ya fue confirmada. Zapier nunca recibe una reservación no disponible.

Pasos del Zap (flujo lineal)

  1. Trigger — Catch Hook
    Recibe JSON con: bookingId, name, email, phone, checkIn, checkOut, roomType (SVM / SFM / SP), guests.
    Las fechas vienen en formato ISO 8601 con offset Guatemala: 2026-06-10T15:00:00-06:00.
  2. Formatter — Lookup Table
    Traduce el código de habitación a nombre completo:
    SVM → Suite Vista Mar | SFM → Suite Frente al Mar | SP → Suite Presidencial
  3. Formatter — Date/Time
    Formatea checkIn → DD/MM/YYYY (para el correo y Sheets).
  4. Formatter — Date/Time
    Formatea checkOut → DD/MM/YYYY.
  5. Google Calendar → Create Detailed Event
    Crea el evento en el calendario "hotel".
    Título: Reservación - {roomType} - {name}
    Zona horaria: America/Guatemala
    Guarda el calendar_event_id generado por Google para uso en n8n.
  6. Gmail → Send Email
    Envía correo de confirmación al huésped (email) con nombre completo de la suite y fechas formateadas.
  7. Google Sheets → Create Spreadsheet Row
    Registra la reserva en la hoja "Reservaciones" con todas las columnas del esquema (incluyendo status = confirmed y el calendar_event_id del paso 5).

6. Workflows de n8n

n8n corre como segundo contenedor en docker-compose (puerto 5678). Los workflows se importan desde los archivos JSON en n8n-workflows/. Cubren el ciclo de vida completo de la reservación: recordatorios, cancelación, modificación y reportes al admin.

Workflow A — Recordatorio 24 horas antes reminder-24h.json

Trigger

Schedule cron: 0 14 * * *
(8:00 AM Guatemala, UTC-6)

Lógica

Lee Sheets → filtra donde check_in == mañana AND status == confirmed AND reminder_24h != TRUE → envía correo con link de cancelación → marca reminder_24h = TRUE.

Workflow B — Recordatorio 2 horas antes reminder-2h.json

Trigger

Schedule: cada 15 minutos.

Lógica

Lee Sheets → filtra status == confirmed AND reminder_2h != TRUE AND check_in ∈ [ahora, ahora+2h] → envía email "Tu check-in es en 2 horas" → marca reminder_2h = TRUE.

El flag reminder_2h garantiza exactamente un correo por reservación sin importar cuántas veces se ejecute el trigger en la ventana de 2 h.

Workflow C — Cancelación de reserva cancelacion.json

Trigger: GET /webhook/cancel-booking?id={booking_id} (link incluido en el email de recordatorio 24h) │ ▼ Google Sheets → buscar fila por booking_id │ ┌────┴────┐ Encontrado No encontrado + confirmed │ └── HTML: "Enlace no válido o ya procesado" │ ▼ Google Calendar → eliminar evento │ ▼ Google Sheets → status = "cancelled" │ ▼ Gmail → correo de cancelación al huésped │ ▼ HTML: "Tu reservación ha sido cancelada exitosamente"

Workflow D — Modificación de fechas modificacion.json

Trigger

Form Trigger en /form/modify-booking.
Campos: ID de Reservación, nueva fecha check-in, nueva fecha check-out.

Lógica

Valida fechas → busca en Sheets → si confirmed: elimina evento anterior en Calendar, crea nuevo evento, actualiza Sheets (fechas + calendar_event_id + status=modified + reset de flags) → envía correo.

Workflow E — Reporte mensual reporte-mensual.json

Trigger

Schedule cron: 0 13 1 * *
(7:00 AM Guatemala, día 1 de cada mes)

Lógica

Lee todas las filas de Sheets → filtra reservas del mes anterior → calcula totales por suite, noches y ingresos (Q) → envía tabla HTML al ADMIN_EMAIL.

7. Configuración del Agente de IA

7.1 Modelo

Se usa claude-sonnet-4-6 (Anthropic Claude API). El agente corre enteramente en el servidor; la API key nunca llega al navegador.

7.2 Sistema RAG (Retrieval-Augmented Generation)

El módulo rag.ts implementa un RAG liviano sin base de datos vectorial:

  1. Los archivos fuente de conocimiento son 4 documentos Markdown en src/lib/knowledge/: rooms.md, policies.md, services.md, faq.md.
  2. Cada mensaje del usuario dispara retrieveContext(query), que hace matching por keywords sobre el texto de los .md.
  3. Los fragmentos más relevantes se adjuntan al system prompt antes de cada llamada a Claude.
  4. Para actualizar el conocimiento del agente basta con editar los archivos .md — no hay índice ni embedding que reconstruir.

7.3 Tool Use

El agente tiene definido exactamente un tool. Claude lo invoca cuando el usuario ha expresado intención clara de hacer una reservación:

// Definición del tool (claude.ts)
{
  name: "show_booking_form",
  description: "Muestra el formulario de reservación al cliente cuando está listo para reservar",
  input_schema: {
    type: "object",
    properties: {},
    required: []
  }
}

El frontend (ChatMessage.tsx) detecta un bloque tool_use con name === "show_booking_form" en la respuesta de la API y renderiza el componente <BookingForm /> en línea dentro del historial del chat.

7.4 Historial de conversación

El historial se mantiene en estado React del cliente (useState) y se envía completo al backend en cada turno. No se persiste en base de datos.

9. Despliegue

Local (desarrollo)

# Sin Docker
npm run dev

# Con Docker Compose (app + n8n)
docker-compose up --build

Producción (Azure Container Apps)

  1. Construir imagen: docker build -t azul-horizonte-app .
  2. Push a Azure Container Registry: az acr build ...
  3. Crear Container App Environment en Azure Portal.
  4. Deploy del contenedor app (ingress público, puerto 3000).
  5. Deploy del contenedor n8n (ingress interno, puerto 5678).
  6. Configurar variables de entorno vía Azure Portal o az containerapp update --set-env-vars.
